# SIMTraversal
## Structured Multi-Agent Traversal — Conflict Generation and BELLA Persistence
## Companion Specification v1.0

**Companion to:** PsiCollapse5A.md (Verification Stability Stack — A4, A4′, A5)
**Does not modify:** PsiCollapse5A.md
**Status:** Canonical integration reference for traversal-generated conflict data
**Scope:** TraversalRecord schema, lifecycle isolation, promotion/demotion rules,
voice-based structural filtering, instrumentation, ANCHOR integration invariants

---

## Document Scope

BELLA persists beliefs generated from organic conversation. This document specifies
how beliefs and conflicts generated by structured multi-agent traversal — repeated
passes over the same contradiction using role-differentiated agents — enter and
interact with the BELLA belief graph.

The central claim is operational, not theoretical: repeated structured traversal
over a contradiction generates recurring disagreement pairs. Those pairs are not
prompt noise. They are structured signals that BELLA can persist using ConflictRecord
recurrence chains, and that can eventually produce ANCHOR-qualified invariants.

This document specifies the data structures, lifecycle rules, and invariants required
to make that claim testable.

## Section 1 — Dependency Order

TraversalRecord connects to existing BELLA/OmegaRE machinery. It must not be
implemented before its dependencies exist and are tested.

```
Stage 1 — Required before TraversalRecord can be built:
  ConflictRecord lifecycle       (OmegaRE core)
  A4 observation events          (PsiCollapse5A)

Stage 2 — Traversal layer (this document):
  TraversalRecord schema
  Lifecycle isolation rules
  Promotion/demotion rules
  ANCHOR integration

Rationale: Building TraversalRecord against unimplemented dependencies
means two unverified systems are tested against each other. When something
breaks, the failure source is ambiguous. Implement Stage 1 fully, test it,
then build Stage 2 against working infrastructure.
```

## Section 2 — Core Distinction: CLOSED vs RESOLVED

These two terms are related but not equivalent. Conflating them will produce
incorrect implementations.

```
TraversalRecord.resolution_status = CLOSED
  Meaning: the synthesis gate (e7) produced a closure outcome for this traversal.
  Signal:  the multi-agent pass converged — e7 synthesized a stable resolution
           over the agent outputs.
  Scope:   one traversal instance.

ConflictRecord.status = RESOLVED
  Meaning: the underlying belief tension has been resolved by BELLA/OmegaRE
           organic graph mechanics — mass dynamics, dominance establishment,
           contradiction pressure.
  Signal:  the belief-level conflict is no longer active in the graph.
  Scope:   the graph's persistent conflict registry.
```

**These can diverge in both directions:**

| Traversal | ConflictRecord | Interpretation |
|---|---|---|
| CLOSED | RESOLVED | Normal successful resolution |
| CLOSED | UNRESOLVED | e7 synthesized but beliefs remain in tension |
| LOCKED | RESOLVED | Organic resolution without traversal convergence |
| LOCKED | UNRESOLVED | Persistent structural conflict — primary signal |
| INDETERMINATE | any | Synthesis failed; no traversal signal available |

The LOCKED + UNRESOLVED combination across repeated traversals is the primary
signal that a contradiction is structurally persistent and warrants elevated
ConflictRecord recurrence tracking.

## Section 3 — TraversalRecord Schema

### 3.1 Full Schema

```python
@dataclass
class TraversalRecord:
    # ── Identity ──────────────────────────────────────────────────
    traversal_id:               UUID
    parent_traversal_id:        UUID | None     # links iteration chain
    timestamp:                  datetime        # traversal start time (UTC)

    # ── Input snapshot ────────────────────────────────────────────
    seed_belief_id:             UUID | None     # NULL if fresh query; backfilled at ingest
    role_sequence:              list[str]       # ordered role names as executed
    agent_config_hashes:        dict[str, str]  # role_name → hash(model + system_prompt)
    shared_context_hash:        str             # hash of expand pack fed to all roles

    # ── Output references ─────────────────────────────────────────
    conflict_pair_ids:          list[UUID]      # ConflictRecord IDs generated this traversal
    traversal_beliefs_produced: list[UUID]      # BeliefNode IDs ingested from agent outputs

    # ── Status ────────────────────────────────────────────────────
    traversal_status:           Enum[COMPLETE, INVALID]
    resolution_status:          Enum[CLOSED, LOCKED, INDETERMINATE]
```

### 3.2 Field Notes

**`parent_traversal_id`** — links successive traversals over the same seed into an
iteration chain. Query `WHERE parent_traversal_id = X` to retrieve all descendants.
Enables convergence_delta computation (Section 9).

**`seed_belief_id`** — may be NULL at traversal time when the input is a fresh user
query not yet in the graph. Backfilled atomically at ingest. If ingest fails,
`traversal_status` becomes INVALID (see Section 4).

**`role_sequence`** — the actual execution order of roles, not the nominal order.
Required for traversal-order hypothesis testing (Section 10.2). Must be recorded
as run, not as configured.

**`agent_config_hashes`** — per-role hash of model identifier, temperature, and
system prompt. A single hash across all roles is insufficient: role-specific config
changes are invisible without per-role granularity.

**`shared_context_hash`** — hash of the expand pack provided to all roles as graph
context. This is a comparability hash, not a reconstruction key (see Section 4.2).

**`resolution_status = INDETERMINATE`** — assigned when e7 fails to run: agent
timeout, model error, synthesis step crash. The traversal completed e0–e6 but was
never synthesized. INDETERMINATE traversals are excluded from all rate calculations
(Section 9).

## Section 4 — Atomicity and Validity Rules

### 4.1 Write Boundaries

Both list fields are finalized atomically with TraversalRecord commit:

```
conflict_pair_ids:
  Finalized after all agent outputs have been processed and all ConflictRecords
  generated. If post-traversal processing fails partway through, the entire
  TraversalRecord is set to traversal_status = INVALID.

traversal_beliefs_produced:
  Finalized after all agent output beliefs have been ingested into the graph.
  If ingest partially succeeds (some beliefs stored, some not), the entire
  TraversalRecord is set to traversal_status = INVALID.
```

Partial commits are not permitted. A TraversalRecord is either fully committed
(COMPLETE) or fully invalidated (INVALID).

### 4.2 seed_belief_id Backfill

```
If seed is a fresh user query (not yet in graph):
  1. TraversalRecord written with seed_belief_id = NULL, traversal_status = COMPLETE
  2. Ingest runs: query belief created and assigned belief_id
  3. TraversalRecord updated: seed_belief_id = new belief_id (atomic update)
  4. If ingest fails: traversal_status = INVALID (NULL state does not persist)
```

INVALID records are retained in storage for audit purposes but excluded from
all recurrence queries, rate calculations, and lineage traversals.

### 4.3 Replay Comparability vs Reconstruction

`shared_context_hash` provides **replay comparability**, not full replay
reconstruction.

```
What it provides:
  Given two traversals, shared_context_hash equality proves they received
  identical graph context. A mismatch flags the comparison as non-comparable.

What it does not provide:
  If the graph has changed since the original traversal, the original expand
  pack cannot be reconstructed from the hash alone.

Implication:
  Full deterministic replay requires storing the actual expand pack content,
  not just its hash. This storage is optional and deferred. For traversal-order
  hypothesis testing, hash comparability is sufficient: confirm matching
  shared_context_hash before comparing outcomes across traversals.
```

## Section 5 — Lifecycle Isolation

### 5.1 Two Conflict Sources

BELLA operates with two distinct ConflictRecord sources after SIMTraversal is
integrated:

```
source = "organic"    Conflicts arising from natural belief accumulation
                      in conversation. Full participation in all mechanics
                      from creation.

source = "traversal"  Conflicts arising from multi-agent traversal output.
                      Begin in quarantine phase. Graduate to organic via
                      promotion rule (Section 6).
```

### 5.2 Quarantine Phase Behavior

While `source = "traversal"` and not yet promoted:

```
Traversal-sourced ConflictRecords:
  DO NOT count toward HIGH_TENSION thresholds
    → ANCHOR qualification is not blocked by traversal conflicts in quarantine
  DO NOT increment omega_distance for A4′ convergence calculations
    → traversal activity does not corrupt convergence signals
  DO NOT participate in guard-block logic
    → guard fires only on conflicts with established semantic signal
  DO accumulate recurrence_count
    → recurrence across traversals is tracked and queryable
  DO participate in conflict_pair_hash recurrence chains
    → same belief-level conflict appearing across traversals is detected
```

Traversal-produced beliefs (in `traversal_beliefs_produced`) are tagged
`source: traversal` and also excluded from omega_distance calculations until
they reach organic status through normal mass accumulation or traversal promotion.

### 5.3 Extended ConflictRecord Fields

```python
class ConflictRecord:
    # ── Existing fields (unchanged) ──────────────────────────────
    id:               UUID
    belief_a:         UUID
    belief_b:         UUID
    status:           Enum[UNRESOLVED, SUSPENDED, DOMINANT, RESOLVED]
    dominant_belief:  UUID | None
    cycle_count:      int
    tension_level:    Enum[LOW, MEDIUM, HIGH]
    created_at:       int
    last_evaluated:   int
    superseded:       bool          # from PsiCollapse5A
    superseded_by:    UUID | None   # from PsiCollapse5A

    # ── SIMTraversal additions ────────────────────────────────────
    source:           Enum[ORGANIC, TRAVERSAL]  = ORGANIC
    conflict_voice_a: str | None   # role name that produced belief_a side
    conflict_voice_b: str | None   # role name that produced belief_b side
    traversal_id:     UUID | None  # originating traversal (TRAVERSAL source only)
```

## Section 6 — Promotion and Demotion

### 6.1 Promotion Rule

A traversal-sourced ConflictRecord is eligible for promotion to organic status
when all three conditions hold:

```
recurrence_count ≥ N_semantic_threshold (default: 3)
AND
distinct_voice_pairs(conflict_pair_hash) ≥ 2
AND
conflict_pair_hash is identical across all counted occurrences
```

`distinct_voice_pairs` counts distinct `(conflict_voice_a, conflict_voice_b)` tuples
across all ConflictRecords sharing the same `conflict_pair_hash`. A minimum of 2
distinct voice pairs means the same belief-level conflict has surfaced via at least
two different role paths — evidence it is not purely structural.

**Promotion smoothing:** at most K = 2 promotions are processed per re_cycle.
This prevents simultaneous bulk promotion from spiking omega_distance when many
traversal conflicts reach threshold at the same time.

**Promotion is a ConflictRegistry mutation event.** It triggers the ANCHOR cascade
rule (Section 8) for all beliefs associated with the promoted ConflictRecord.

### 6.2 Post-Promotion Behavior

After promotion to `source = ORGANIC`:

```
Full participation in guard-block logic
Eligible for ANCHOR qualification mechanics
Included in omega_distance calculations
Subject to normal ConflictRecord lifecycle (escalation, dominance, resolution)
```

### 6.3 Demotion Rule

A promoted ConflictRecord is eligible for demotion if subsequent traversals
indicate it was structural rather than semantic:

```
Condition:
  In the next P = 3 traversals after promotion,
  the conflict resolves (CLOSED) in ≥ M = 2 of those traversals
  OR
  resolution_status varies (CLOSED in some, LOCKED in others)

  Role order variation is NOT required for demotion evaluation.
  Resolution outcomes are counted regardless of role_sequence.

Eligibility constraint:
  Demotion applies only to promoted ConflictRecords with
  status ∈ {UNRESOLVED, SUSPENDED}.
  Records that have reached RESOLVED or DOMINANT through organic lifecycle
  mechanics are not eligible for demotion — organic lifecycle takes precedence.
```

**Demotion record handling:**

```python
def demote(record: ConflictRecord, graph: BeliefGraph) -> None:
    # Mark existing organic record as superseded
    record.superseded = True
    record.superseded_by = new_record.id   # assigned below

    # Create replacement traversal-source record
    new_record = ConflictRecord(
        id               = new_uuid(),
        belief_a         = record.belief_a,
        belief_b         = record.belief_b,
        status           = UNRESOLVED,
        source           = TRAVERSAL,
        recurrence_count = 0,              # reset
        conflict_voice_a = record.conflict_voice_a,
        conflict_voice_b = record.conflict_voice_b,
        ...
    )
    graph.conflict_registry.add(new_record)

    log_audit(f"CONFLICT_DEMOTED [{record.id}] → [{new_record.id}]")
```

The superseded record is retained for lineage audit. It does not participate in
active conflict mechanics.

### 6.4 Constants

```
N_semantic_threshold = 3     # recurrences required for promotion
M                    = 2     # resolutions out of P that trigger demotion
P                    = 3     # traversal window for demotion evaluation
K                    = 2     # max promotions per re_cycle (smoothing)
```

All are tunable. Defaults are symmetric: promotion requires 3 recurrences;
demotion triggers if 2 of the next 3 traversals resolve the conflict.

## Section 7 — Voice-Based Structural Filtering

### 7.1 conflict_pair_hash Definition

```
conflict_pair_hash = hash(min(belief_a_id, belief_b_id), max(belief_a_id, belief_b_id))
```

The hash is computed over **belief IDs, not voice IDs**. This is a hard requirement.
If voice IDs are included in the hash, the same semantic conflict surfaced by
different roles produces different hashes, and the recurrence chain breaks.

The canonical form uses sorted IDs to make the hash order-independent: the conflict
between belief X and belief Y is the same regardless of which side is labeled `a`.

### 7.2 Classification Rules

```
Structural classification:
  conflict_pair_hash is constant across recurrences
  AND distinct_voice_pairs(conflict_pair_hash) == 1
  AND resolution_status is consistent across traversals

  Interpretation: same roles always disagree about the same beliefs,
  always with the same outcome. The tension is a property of role design,
  not of the beliefs themselves.
  Action: keep as source=TRAVERSAL, do not promote.

Semantic classification:
  conflict_pair_hash is constant across recurrences
  AND distinct_voice_pairs(conflict_pair_hash) ≥ 2

  Interpretation: the same belief-level conflict surfaces via different role
  paths. The tension is in the beliefs, not in the role assignment.
  Action: eligible for promotion per Section 6.1.

Exception — constant voice pair with varying resolution:
  conflict_pair_hash is constant
  AND distinct_voice_pairs == 1
  AND resolution_status varies across traversals

  Interpretation: same roles always contest the same beliefs, but the outcome
  is order-dependent. This may indicate a genuine semantic conflict that only
  certain roles are positioned to detect (e.g., a falsification role and a
  synthesis role are the only ones examining this particular claim type).
  Action: classify as REVIEW_REQUIRED, not structural. Do not auto-demote.
  Surface in bellamem audit output for manual review.
```

### 7.3 "Resolution status varies" Definition

Resolution status varies across traversals when the set of `resolution_status`
values in `TraversalRecord` for traversals sharing the same `conflict_pair_hash`
contains more than one distinct value from {CLOSED, LOCKED}. INDETERMINATE
traversals are excluded from this check.

## Section 8 — ANCHOR Integration

### 8.1 Promotion as ConflictRegistry Mutation

Promotion of a traversal conflict to organic status is treated as a
ConflictRegistry mutation event. The A5 cascade rule (PsiCollapse5A, Section 3.3)
fires on this event:

```python
def on_conflict_promoted(record: ConflictRecord, graph: BeliefGraph) -> None:
    # Treat exactly like a new ConflictRecord for cascade purposes
    affected_entity_ids = extract_entities(record.belief_a, record.belief_b, graph)

    for entity_id in affected_entity_ids:
        hu_anchors = graph.get_beliefs_where(
            scope=ScopeEnum.HISTORICAL_UNIVERSAL,
            status=BeliefStatus.ANCHOR,
            references_entity=entity_id
        )
        for belief in hu_anchors:
            invalidate_hcc_cache(belief.id)
            passed, reason = historical_contradiction_check(belief, graph)
            if not passed:
                belief.status = BeliefStatus.ACTIVE
                belief.anchor_blocked = True
                belief.anchor_blocked_reason = reason
                log_audit(f"ANCHOR_DEMOTED_BY_PROMOTION [{belief.id}]: {reason}")
```

### 8.2 Mid-Cycle ANCHOR Qualification Block

Promotion events that occur during re_cycle invalidate the ANCHOR qualification
cache and block new ANCHOR completion for affected beliefs until the next
re_cycle begins:

```
Invariant:
  If a promotion event fires for ConflictRecord C during re_cycle t,
  then for all beliefs referenced by C:
    - historical_contradiction_check cache is invalidated
    - Any ANCHOR qualification that passed earlier in cycle t is voided
    - Re-qualification must occur in cycle t+1 or later

Rationale:
  Without this rule, a belief can pass the ANCHOR gate early in a cycle,
  then have a newly-promoted organic conflict open against it in the same
  cycle, resulting in a Frozen Anchor (BAD_ATTRACTOR_TYPE_3) created
  silently by the promotion event.
```

### 8.3 Traversal-Origin ANCHOR Candidates

Beliefs in `traversal_beliefs_produced` follow normal ANCHOR qualification
mechanics once they accumulate sufficient organic mass. Their traversal origin
is recorded in `traversal_beliefs_produced` for audit and ANCHOR emergence
rate tracking (Section 9), but does not affect qualification logic.

## Section 9 — Instrumentation

Five instruments are required to demonstrate traversal usefulness empirically.
Three supplementary signals close measurement gaps.

### 9.1 Conflict Pair Resolution Rate

```
For each conflict_pair_hash:

  resolution_rate = resolved_traversal_count / total_traversal_count

  where resolved_traversal_count = count of TraversalRecords with
    conflict_pair_hash in conflict_pair_ids
    AND resolution_status = CLOSED
    AND traversal_status = COMPLETE

Interpretation:
  High rate (≥ 0.8): conflict pair resolves consistently → DOMINANT candidate
  Low rate (≤ 0.2): conflict pair never resolves → genuine structural tension,
                    surface to user
  Variable rate (0.3–0.7): order-dependent or context-dependent → flag for
                            voice filtering review
```

### 9.2 φ⁰ Closure Rate per Seed Belief

```
For each seed_belief_id:

  closure_rate(N) = CLOSED_count(traversals 1..N) / N

  where CLOSED_count counts TraversalRecords with:
    seed_belief_id = this belief
    resolution_status = CLOSED
    traversal_status = COMPLETE

INDETERMINATE traversals are excluded from N and from CLOSED_count.
```

### 9.3 Convergence Delta

```
For each seed_belief_id:

  baseline_rate     = CLOSED_count(traversals 1..B) / B     (B = 3)
  closure_rate(N)   = CLOSED_count(traversals 1..N) / N
  convergence_delta = closure_rate(N) - baseline_rate

Interpretation:
  convergence_delta > 0 and increasing: traversal history is improving φ⁰
    closure — BELLA graph state is feeding back into generation
  convergence_delta ≈ 0: traversal history has no measurable effect
  convergence_delta < 0: traversal is making closure harder — investigate
    whether traversal-produced beliefs are introducing conflicting context

Note on baseline stability:
  baseline_rate with B=3 is a single ratio from 3 binary observations
  (values: 0, 0.33, 0.67, 1.0). For seed beliefs with few traversals,
  baseline_rate is sensitive to early outcomes. Interpret convergence_delta
  for N < 10 with caution.
```

### 9.4 ANCHOR Emergence Rate

```
Tracked separately:
  traversal_anchor_count  = ANCHOR beliefs with id ∈ any traversal_beliefs_produced
  organic_anchor_count    = ANCHOR beliefs with no traversal provenance

Rate per session:
  traversal_anchor_rate = new traversal-origin ANCHORs / session
  organic_anchor_rate   = new organic ANCHORs / session

Interpretation:
  If traversal_anchor_rate > 0: traversal is producing beliefs that stabilize
    into long-term invariants. Traversal is doing useful epistemic work.
  If traversal_anchor_rate = 0 across many sessions: traversal beliefs are
    not surviving to ANCHOR threshold. Either they are being pruned, or
    they are not accumulating sufficient recurrence.
```

### 9.5 Guard Block Attribution

```
For each guard block event:
  record:
    blocking_conflict_source: ORGANIC | TRAVERSAL
    override_type:            ACKNOWLEDGED | SILENT | NO_OVERRIDE

  ACKNOWLEDGED: user explicitly confirmed awareness and proceeded
  SILENT:       user bypassed guard without acknowledgment (likely false positive)
  NO_OVERRIDE:  user accepted the block

guard_correction_rate (traversal) =
  SILENT_override_count(traversal-source blocks) /
  total_traversal_source_blocks

Interpretation:
  High correction rate (> 0.4): traversal conflicts are generating false
    positives — structural noise reaching guard despite quarantine
  Low correction rate (< 0.1): traversal-sourced blocks are accurate
```

### 9.6 Replay Diff (Supplementary)

```
Given two traversals T1 and T2 where:
  shared_context_hash(T1) == shared_context_hash(T2)
  agent_config_hashes(T1) == agent_config_hashes(T2)

Replay diff:
  symmetric_diff(conflict_pair_ids(T1), conflict_pair_ids(T2))

If symmetric_diff is empty: traversal is deterministic for these inputs.
If symmetric_diff is non-empty: traversal is non-deterministic. Investigate
  whether role_sequence differed or whether LLM temperature is non-zero.
```

## Section 10 — Traversal Order Hypothesis

### 10.1 The Hypothesis

Does traversal order — the sequence in which roles are applied — systematically
affect φ⁰ closure outcomes?

This is an open empirical question. SIMTraversal.md specifies the data structures
needed to test it. It does not assert the answer.

### 10.2 Test Design

```
Required data:
  TraversalRecords for the same seed_belief_id with different role_sequence values
  Same shared_context_hash (same graph context)
  Same agent_config_hashes (same role configurations)

Test:
  Group traversals by role_sequence.
  Compare resolution_status distribution across groups.

  If resolution_status distribution is significantly different between
  role_sequence groups → traversal order affects outcomes.

  If distribution is uniform across role_sequence groups → traversal order
  does not affect outcomes; role sequence may be treated as configurable
  but not structurally constrained.
```

### 10.3 Implication for Spec Evolution

If traversal order is demonstrated to affect outcomes systematically, a future
document may introduce ordering constraints on role_sequence. Until that
demonstration exists, `role_sequence` is recorded for analysis but not constrained
by this specification.

## Section 11 — Invariant Summary

```
I1 — Dependency order
  TraversalRecord must not be built before ConflictRecord lifecycle and
  A4 observation events are fully implemented and tested.

I2 — Atomic commit
  conflict_pair_ids and traversal_beliefs_produced are finalized atomically
  with TraversalRecord commit. Partial failure sets traversal_status = INVALID.
  INVALID records are retained for audit but excluded from all queries.

I3 — CLOSED ≠ RESOLVED
  TraversalRecord.resolution_status = CLOSED and ConflictRecord.status = RESOLVED
  are distinct signals that may diverge. They must not be conflated in
  implementation or instrumentation.

I4 — Quarantine isolation
  Traversal-sourced ConflictRecords in quarantine do not affect HIGH_TENSION
  thresholds, omega_distance, or guard logic.

I5 — Promotion smoothing
  At most K = 2 traversal ConflictRecords promote to organic per re_cycle.

I6 — Promotion is a ConflictRegistry mutation
  Promotion triggers the A5 cascade rule. ANCHOR re-evaluation fires for all
  beliefs associated with the promoted ConflictRecord.

I7 — Mid-cycle ANCHOR block
  A promotion event during re_cycle t voids ANCHOR qualifications for affected
  beliefs that completed earlier in cycle t. Re-qualification occurs in cycle t+1.

I8 — Demotion scope
  Demotion applies only to promoted ConflictRecords with status ∈
  {UNRESOLVED, SUSPENDED}. Records that reached RESOLVED or DOMINANT through
  organic mechanics are not eligible. Organic lifecycle takes precedence.

I9 — Demotion record handling
  Demotion sets superseded = True on the existing organic record and creates
  a new ConflictRecord with source = TRAVERSAL and recurrence_count = 0.
  The superseded record is retained for audit.

I10 — conflict_pair_hash over belief IDs
  conflict_pair_hash is computed over sorted (belief_a_id, belief_b_id).
  Voice IDs must not be included. Hash must be order-independent.

I11 — INDETERMINATE exclusion
  INDETERMINATE traversals are excluded from all rate calculations:
  closure_rate, baseline_rate, convergence_delta, and replay diff.

I12 — Replay comparability boundary
  shared_context_hash proves input comparability. It does not enable
  reconstruction of the original expand pack if the graph has changed.
  Full reconstruction requires storing expand pack content (deferred).
```
